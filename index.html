The Recommendation (60-second version)

The fleet's strongest opportunity is condition-based monitoring. 64% of classified failure modes (32 of 50) are Pattern C -- gradual degradation detectable by VIMS with >= 65% predictability. A VIMS threshold alert system (no ML required) can catch the majority of foreseeable breakdowns. Additionally, 22.5% of events recur within 7 days -- post-repair monitoring for the 3 high-infant-rate failure modes adds further value. Start with VIMS alerts, then build a survival model, then address root causes. Details in the Strategy tab.

Key Metric Explanations

C-Index: 0.607 (Current Model)

Measures whether the model correctly ranks which truck will fail first. A C-index of 0.50 = coin flip (random). The current model at 0.607 gets the priority order right only 60.7% of the time -- wrong 4 out of 10. A maintenance planner using this would schedule the wrong truck first nearly half the time.

Health Score Discrimination: ~1 Point

Average score before breakdowns: 68.4. Average during normal operation: 69.5. On a 100-point scale, 1.1 points is noise -- you cannot distinguish a truck about to fail from a healthy one. Worse, component-level scores are inverted: trucks score 10-25 points HIGHER before breakdowns, likely because recent repairs temporarily boost scores.

Production Loss Rate

Dollar figures should use Amman's actual cost per hour of unplanned truck downtime. Industry benchmarks range $2,000-$15,000/hr for mining haul trucks depending on commodity prices and mine productivity.

How was this data calculated? (click to expand methodology)

What counts as a "breakdown"?

A breakdown is a record in the AMT (Asset Management Tool) shift log where BREAKDOWN = TRUE. This is a field set by the maintenance team when logging an event -- it means the truck had an unplanned stop that required maintenance intervention.

We only included breakdowns with >= 4 hours of downtime (ACTUALDOWNHOURS >= 4). This filters out quick inspections, minor adjustments, and false entries. Events under 4 hours are excluded from all metrics in this explorer.

Period: June 2024 to April 2026 (22 months). Fleet: Only CAT 793 trucks (equipment ID starting with "HT", model containing "793").

How each metric is calculated

MetricWhat it meansHow it's calculated
Events Number of breakdown records matching this failure mode COUNT(*) where BREAKDOWN=TRUE, ACTUALDOWNHOURS >= 4, EVENTDESCRIPTION matches, and equipment is HT% / 793 model
Trucks Affected How many different trucks experienced this failure COUNT(DISTINCT TRUCK_ID) -- tells you if this is a fleet-wide problem (many trucks) or truck-specific (few trucks, many events)
Avg Down Hours Average duration of each breakdown event in hours AVG(ACTUALDOWNHOURS) across all events of this type. Note: some events have ACTUALDOWNHOURS = 0 (data quality issue, excluded by our >= 4h filter)
Total Down Hours Sum of all downtime caused by this failure mode SUM(ACTUALDOWNHOURS). This is the cost impact metric -- higher = more production lost
Infant Mortality % What percentage of these events occurred within 7 days of a PREVIOUS breakdown on the same truck AND same system For each event, we look at the previous event on the same truck + same component system using:
LAG(event_timestamp) OVER (PARTITION BY truck_id, system ORDER BY event_timestamp)
If DATEDIFF(day, previous_event, this_event) <= 7, it's counted as infant mortality.
Infant % = infant events / events with a previous event * 100

Important caveat: This measures same-SYSTEM recurrence, not same-DESCRIPTION. A truck with an engine coolant leak fixed on Monday, then an engine oil pressure failure on Friday counts as infant mortality (same system: Engine, within 7 days) even though the specific failure is different. This is a proxy -- it likely overestimates true repair-failure rate by 10-20%.
VIMS Predictable % What percentage of these breakdowns had at least 1 VIMS health alarm in the 7 days BEFORE the breakdown For each breakdown event, we count VIMS alerts from the SILVER_HEALTH_ALARM_EVENT table where:
EQUIPMENT_ID = truck_id AND ACTIVATION_DATE >= event_date - 7 days AND ACTIVATION_DATE < event_date
If count > 0, the event is "VIMS predictable."
VIMS % = events with >= 1 alert / total events * 100

Important caveat: VIMS alerts are not specific to the component that failed. A truck might have brake temperature alerts before an engine failure. The alert existed, but it may not be causally related to this specific failure mode. This metric shows correlation, not causation.
Avg VIMS When Present When a breakdown DID have VIMS alerts, how many alerts were there on average in the 7-day window AVG(alert_count) only for events where alert_count > 0. Higher = stronger signal. A value of 1,549 means almost 1,500 VIMS alerts in 7 days -- an alarm storm. A value of 60 means a few quiet alerts.
Catastrophic Count How many events of this type had >= 168 hours (7 days) of downtime COUNT WHERE ACTUALDOWNHOURS >= 168. These are the most severe events -- truck out of service for a week or more. Even one catastrophic event justifies investment in prediction for that failure mode.

How failure modes are classified into RCM patterns

The classification uses two primary metrics:

PatternRuleLogic
F -- Infant MortalityInfant % >= 50% AND not also showing wear-out characteristicsMore than half of events are repeats within 7 days of a prior repair. The repair is the risk.
C -- FatigueVIMS % >= 65% AND Infant % < 50%Good VIMS signal AND repairs hold (low infant rate). Classic gradual degradation with P-F curve.
E -- RandomVIMS % < 50% AND Infant % < 40%Low VIMS signal AND repairs hold. No detectable pattern -- random occurrence.
A -- BathtubInfant % 35-60% AND VIMS % 40-70% AND catastrophic events or high severityShows BOTH infant mortality and wear-out characteristics. Needs dual strategy.
B -- Wear-outKnown component-life-limited items with low infant rateApplied to seals, filters, bearings where the failure correlates with hours/cycles, not repair quality.
D -- Break-inPost-overhaul stabilization pattern (low initial, rising to constant)Rare in this dataset. Applied when a failure mode is concentrated in recently overhauled trucks.

Note: These classifications are heuristic, not definitive. Some failure modes sit on boundaries between patterns. The "predict?" decision is based on whether the VIMS signal is strong enough AND the failure has enough economic impact to justify monitoring investment. A 70% predictable failure with 50 hours total downtime is technically predictable but not worth building a system for.

What "Predict: Yes / Partial / No" means

Data quality notes

  • Spelling variants: The same failure appears under multiple descriptions (e.g., "TRANSMISSION SHIFTING PROBLEM", "TRANSMISI SHIFTING PROBLEM", "TRANS SHIFTING PROBLEM"). These are shown separately because the source data has no normalization. In a production system, these should be consolidated.
  • System classification: Component codes are mapped to systems via GOLD_COMPONENT_GROUP_MAPPING. 6.4% of events have unmapped codes and appear as "Unmapped."
  • Fan-out deduplication (CORRECTED): A single physical breakdown event can generate multiple rows in AMT (one per work order). The Explorer data IS deduplicated using a 4-layer query: GROUP BY (truck, system, failure_mode, timestamp) collapses multi-WO events into 1 physical breakdown. The "Events" column shows physical breakdowns; the "WOs" column shows total work orders. This correction reduced the dataset from 7,610 raw rows to 2,366 unique events and eliminated the false infant mortality signal found in the original analysis.
  • VIMS is not system-specific: The VIMS alert count is for ALL alerts on that truck, not filtered to the specific system that failed. A hydraulic failure preceded by engine VIMS alerts would still show as "VIMS predictable." This overestimates true system-specific predictability.
  • The 4-hour filter: By excluding events < 4 hours, we're looking at significant breakdowns only. This removes ~58% of all breakdown records (quick fixes). If you remove this filter, the infant mortality % drops because many quick fixes are independent events.

Exploratory Data Analysis: Key Findings

~34,300 breakdown events • 146-148 active CAT 793 trucks (158 registered) • 40 months (Jan 2023 - Apr 2026). Primary analysis uses Jan 2024+ (28,707 events, 28 months). VIMS correlation uses June 2024+ (22 months)

Top 5 Findings

1. Health scores are inverted -- trucks score 10-25 points HIGHER before breakdowns. The system directs attention AWAY from at-risk trucks.
2. VIMS 7-day alert volume is the strongest predictor -- 4x higher before breakdowns vs baseline. A simple count threshold outperforms the current ML model.
3. 40% of breakdowns have no VIMS precursor -- bimodal distribution. The ceiling for VIMS-based prediction is ~60-68%.
4. 3.5x regime change in early 2025 -- breakdowns tripled (400 to 1,400+/month). Fleet size unchanged. Cause unknown. Must be resolved before training any model.
5. SOS oil analysis shows no predictive signal -- iron/copper PPM are identical before breakdowns vs fleet average. Sampling may be too infrequent.
